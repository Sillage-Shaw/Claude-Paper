# Project Instructions

This project is a Claude Code research workspace for academic literature survey, paper translation, paper deep reading, and periodic paper tracking.

`CLAUDE.md` defines the shared project rules for file layout, output paths, file creation, append behavior, and final response format.

Skill-specific reasoning and workflow details should stay in each skill's `SKILL.md`.

If any skill instruction conflicts with this file, follow this `CLAUDE.md` unless the user explicitly specifies another path or rule.

---

## 1. Core Tools and Skills

Use these skills for academic research tasks:

* `field-survey`: literature survey, field research, research landscape, paper table, reading roadmap.
* `academic-translation`: strict paragraph-by-paragraph academic translation.
* `paper-deep-reading`: deep analysis of a selected paper, including method, assumptions, experiments, limitations, and research ideas.
* `arxiv-watch`: periodic tracking of new papers in a research field.
* `paper-search`: paper search, metadata lookup, PDF download, and paper reading.

For literature survey and paper search tasks, do not rely only on model memory. Use `paper-search` when available.

Preserve standard English technical terms when appropriate, especially method names, model names, dataset names, benchmark names, and technical acronyms.

## 3. Output Path Rules

### 3.1 PDF files

Downloaded or manually collected papers should be saved under:

```text
./papers/<field-slug>/<paper-file>.pdf
```

Examples:

```text
./papers/action-chunking/act.pdf
./papers/vla/openvla.pdf
./papers/diffusion-policy/diffusion-policy.pdf
```

Use the research field or topic as `<field-slug>`.

---

### 3.2 Field-level outputs

Outputs about a research field, direction, or topic should be saved under:

```text
./note/fields/<field-slug>/
```

Common files:

```text
./note/fields/<field-slug>/survey.md
./note/fields/<field-slug>/paper-table.md
./note/fields/<field-slug>/reading-roadmap.md
./note/fields/<field-slug>/open-problems.md
```

Use this path for outputs from `field-survey`.

Examples:

```text
./note/fields/vla/survey.md
./note/fields/action-chunking/paper-table.md
```

---

### 3.3 Single-paper outputs

All outputs related to one specific paper should be saved under:

```text
./note/papers/<paper-slug>/
```

Common files:

```text
./note/papers/<paper-slug>/metadata.md
./note/papers/<paper-slug>/translation.md
./note/papers/<paper-slug>/terminology.md
./note/papers/<paper-slug>/translation-notes.md
./note/papers/<paper-slug>/deep-reading.md
./note/papers/<paper-slug>/qa.md
```

Use this path for outputs from `academic-translation` and `paper-deep-reading`.

The `deep-reading.md` output must follow the structure defined in `./templates/paper-note.md`.

Examples:

```text
./note/papers/act/translation.md
./note/papers/act/deep-reading.md
./note/papers/openvla/terminology.md
```

---

### 3.4 ArXiv watch outputs

Outputs generated by `arxiv-watch` should be saved under:

```text
./watch/arxiv/<field-slug>/<date>.md
```

Examples:

```text
./watch/arxiv/vla/2026-05-29.md
./watch/arxiv/action-chunking/2026-05-29.md
```

## 5. File Creation Rule

Create files only when they are needed.

Examples:

* `academic-translation` normally creates or updates:

  * `./note/papers/<paper-slug>/translation.md`
  * `./note/papers/<paper-slug>/terminology.md`
  * optionally `./note/papers/<paper-slug>/translation-notes.md`

* `paper-deep-reading` normally creates or updates:

  * `./note/papers/<paper-slug>/deep-reading.md` — must follow the structure defined in `./templates/paper-note.md`
  * optionally `./note/papers/<paper-slug>/qa.md` when the user asks specific deep questions

* `field-survey` normally creates or updates:

  * `./note/fields/<field-slug>/survey.md`
  * optionally `paper-table.md`, `reading-roadmap.md`, or `open-problems.md` when useful or requested

Do not pre-create empty files.

Do not create `deep-reading.md` during translation unless the user explicitly asks for deep reading.

Do not create `translation.md` during deep reading unless the user explicitly asks for translation.

## 6. Append and Overwrite Rules

Do not overwrite existing research notes unless the user explicitly asks.

Prefer append mode for:

```text
translation.md
deep-reading.md
qa.md
```

### 6.1 Translation append rule

For the same paper, append all translated sections to:

```text
./note/papers/<paper-slug>/translation.md
```

Do not create separate files such as:

```text
abstract-translation.md
introduction-translation.md
method-translation.md
```

unless the user explicitly asks.

Before appending a new translation batch, insert:

```markdown
---

# Translation Batch: <section-or-range>

Source: <paper path, section name, page range, or pasted text>

---
```

The terminology table should be saved or updated at:

```text
./note/papers/<paper-slug>/terminology.md
```

### 6.2 Deep reading append rule

For the same paper, append deep reading notes to:

```text
./note/papers/<paper-slug>/deep-reading.md
```

Before appending a new deep reading batch, insert:

```markdown
---

# Deep Reading Batch: <question-or-topic>

Source: <paper path, translation note, section, or user question>

---
```

### 6.3 Q&A append rule

For paper-specific questions, append Q&A entries to:

```text
./note/papers/<paper-slug>/qa.md
```

Use this structure:

```markdown
---

# Question: <user question>

## Short Answer

## Detailed Analysis

## Evidence or Source

## Remaining Uncertainty
```

## 7. Slug Rules

Use lowercase kebab-case for `<field-slug>` and `<paper-slug>`.

Examples:

```text
Vision-Language-Action -> vla
Action Chunking -> action-chunking
Action Chunking Transformer -> act
Diffusion Policy -> diffusion-policy
OpenVLA -> openvla
RAG Evaluation -> rag-evaluation
```

For paper slugs:

1. Prefer a short, stable paper name.
2. Use a widely known acronym when available.
3. If no acronym exists, use a shortened kebab-case paper title.
4. If unsure, choose a reasonable slug and mention it in the final response.

---

## 8. Path Priority Rule

When saving files, follow this priority order:

1. If the user explicitly specifies a path, follow the user-specified path.
2. Otherwise, follow the file layout in this `CLAUDE.md`.
3. If a skill document specifies a conflicting path, follow this `CLAUDE.md`.
4. If the correct path is unclear, choose the closest path under `./note/`, `./papers/`, or `./watch/`, and mention the chosen path in the final response.

## 9. Safety Rules

Do not modify these files or directories unless the user explicitly asks:

```text
.env
.env.*
.vscode/
.git/
.claude/settings*
```

Do not use destructive commands unless the user explicitly approves.

Avoid overwriting existing research notes. Prefer appending with a clear separator.

## 10. Final Response Rule

After creating or modifying research files, always tell the user:

1. what was created or updated
2. exact file path
3. whether content was created or appended
4. what the next recommended step is

Example:

```text
已完成 ACT 的 Method 部分翻译，并追加到：

./note/papers/act/translation.md

术语表已更新：

./note/papers/act/terminology.md

建议下一步使用 paper-deep-reading 分析 Method 的核心假设和实验支撑。
```
## 11. Session Notes

Session notes are used to summarize high-intensity AI interaction processes, such as debugging, environment setup, tool installation, model configuration, or GitHub project onboarding.

Outputs generated by `session-note` should be saved under:

```text
./note/sessions/<session-slug>.md
```

Examples:

```text
./note/sessions/claude-code-deepseek-vscode-setup.md
./note/sessions/paper-search-mcp-installation.md
./note/sessions/github-project-env-setup.md
./note/sessions/node-nvm-vscode-terminal-fix.md
```

Use `session-note` for:

* debugging notes
* environment setup notes
* installation records
* configuration records
* bug-fix summaries
* AI-assisted troubleshooting summaries
* reusable operation notes

Do not record the full conversation transcript by default.

Focus on reusable knowledge:

* background / goal
* problem symptoms
* error messages
* environment information
* root cause
* failed attempts
* final solution
* verification method
* reusable commands or configuration
* lessons learned
* follow-up tasks
